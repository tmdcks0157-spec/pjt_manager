Playwright로 더미 데이터를 삽입하며 취약점/개선점을 점검하고, 발견된 보안·안정성 이슈를 하드닝.
RLS 정책을 버전 관리로 박제하고, 세션 가드·입력 길이 제한·레이스 컨디션 완화 등을 적용.

> 참고: 작업 중 원격(origin/main)이 로컬보다 크게 앞서 있던(CRM + 칸반 리팩토링) 상황을 발견 →
> 리셋 후 최신 구조에 보안 작업을 재적용하는 방식으로 처리.

## 2. RLS 정책 버전화 (최우선)

- 파일: `supabase/migrations/0001_enable_rls.sql`, `0002_dedupe_policies.sql`
- 이 앱은 서버 없이 브라우저에서 anon key로 Supabase 직접 접근 → **보안이 전적으로 RLS에 의존**하는데 마이그레이션 파일이 repo에 없었음
- 6개 소유 테이블(projects/columns/tasks/checklist_items/calendar_events/posts)에 `auth.uid() = user_id` 정책을 idempotent하게 적용
- 검증 결과: **원래도 RLS가 있었음**(실제 구멍 없었음) → 0001로 버전화 + 0002로 중복 정책 정리
- 최종: 각 테이블 단일 정책 `Users manage own rows` (ALL, authenticated). 라이브 DB 반영 완료

## 3. 세션 가드 — user!.id 비널 단언 제거

- 신규: `src/lib/auth.ts` `requireUserId()` — 세션 없으면 명확한 에러 + 로그인 유도
- `const { data: { user } } = await supabase.auth.getUser()` ... `user!.id` 패턴 **15곳** 교체
  - calendar(2), dashboard(1), today(1), projects/[id](5), CreateTaskModal(2), kanban/TaskModal(1), hooks/useActivities(1), hooks/useCRM(2)
- 세션 만료 상태에서 INSERT 시 `user`가 null → 크래시하던 문제 제거

## 기술 메모

- Supabase는 다중 permissive 정책을 OR로 합침 → 중복 정책은 무해하나 혼동 방지 위해 정리(0002)
- RLS 마이그레이션은 Supabase SQL Editor에 **수동 실행해야 반영**됨 (자동 적용 아님)
- `do $$ ... $$` 블록 / `drop policy`는 "Success. No rows returned"가 정상 성공
- 로컬 QA 스크립트는 자격증명을 env로만 주입 (`$env:PM_EMAIL=...; $env:PM_PASS=...; node dummy.cjs`)
