## 0. Executive Summary

| 관점 | 내용 |
|------|------|
| **Problem** | 개인 생산성 도구로 만든 칸반 앱이 충분히 쓸 만하지만, (1) 프로젝트가 늘면 태스크를 찾기 어렵고, (2) 소방/설비 실무(견적·발주·현장점검)의 반복 행정 업무는 여전히 엑셀/카톡/수기로 분산되어 있다. |
| **Solution** | pjt_manager에 "찾기·우선순위·놓침 방지" 3축 기능을 더해 일일 사용 밀도를 높이고, 칸반 엔진을 재사용해 실무용 신규 모듈("현장 점검 체크리스트")로 확장한다. |
| **기능/UX 효과** | 전역 검색·우선순위 배지·마감 알림으로 "여러 프로젝트를 한눈에 통제"하는 경험 완성. 현장 점검 모듈로 개인 도구 → 업무 도구 전환. |
| **Core Value** | "흩어진 일을 한 곳에서 놓치지 않고 끝낸다" — 개인이든 현장이든 동일한 칸반 멘탈모델로. |

**최종 추천 (근거는 8장 참조)**
- pjt_manager 추가기능 TOP 3: ① 전역 태스크 검색 ② 우선순위 배지+필터 ③ 마감일 알림 배지
- 신규 프로그램 TOP 1: **현장 점검 체크리스트 앱** (소방/설비 실무 — 칸반/체크리스트 엔진 재사용도 최고)

## 1. Opportunity Solution Tree (OST)

> Teresa Torres 프레임워크. Desired Outcome → Opportunities(사용자 니즈) → Solutions → Experiments.

### Desired Outcome (목표 성과)
**"사용자가 매일 앱을 열고, 자기 일/업무를 놓치지 않고 끝낸다"** (일일 활성 사용 + 태스크 완료율)

### Opportunity 1 — "프로젝트가 늘면 원하는 태스크를 못 찾는다"
- **Pain**: 프로젝트별 칸반은 격리되어 있어, "그 거래처 건 어디 적었지?"를 찾으려면 보드를 일일이 연다.
- Solutions:
  - **S1.1 전역 태스크 검색** (모든 프로젝트 통합, 제목·설명·태그) ⭐
  - S1.2 태그 기반 필터/그룹
  - S1.3 칸반 → 리스트/테이블 뷰 전환 (정렬·일괄 스캔)
- Experiment: 검색창에 입력 시 0.5초 내 결과 / 검색 사용 후 7일 리텐션 비교

### Opportunity 2 — "무엇을 먼저 해야 할지 한눈에 안 보인다"
- **Pain**: 모든 태스크가 시각적으로 동등. 긴급/중요 구분이 안 됨. (스키마엔 `priority` 필드 이미 존재, UI 미구현)
- Solutions:
  - **S2.1 우선순위 배지** (낮음/보통/높음/긴급, 색상) ⭐ — *백엔드 준비 완료, UI만 추가*
  - S2.2 우선순위별 필터/정렬
  - S2.3 마감 임박 하이라이트
- Experiment: 배지 도입 후 "긴급" 태스크 평균 처리 소요시간 단축 측정

### Opportunity 3 — "마감을 놓친다"
- **Pain**: 앱을 안 열면 마감을 모름. 캘린더는 능동적으로 봐야만 보임.
- Solutions:
  - **S3.1 탭 타이틀 알림 배지** (기한초과 N건) ⭐ — 무료, 즉시 구현
  - S3.2 브라우저 푸시 알림 (Web Push)
  - S3.3 주간 리포트 (이번 주 완료/지연 요약)
- Experiment: 배지 노출군의 기한초과 태스크 비율 감소

### Opportunity 4 — "실무(견적/발주/현장점검)는 앱 밖에서 따로 한다"
- **Pain**: POSCO E&C·KOGAS 거래처 업무는 엑셀/카톡/수기로 분산. 개인 도구와 단절.
- Solutions:
  - **S4.1 현장 점검 체크리스트 모듈** (칸반/체크리스트 엔진 재사용) ⭐
  - S4.2 견적/발주 추적 보드 (칸반 status = 견적→발주→납품)
  - S4.3 거래처 CRM (연락처+이력)
- Experiment: 실제 현장 1건을 앱으로 점검 → 엑셀 대비 소요시간/누락률 비교

> ⭐ = 후속 우선순위 매트릭스에서 상위로 선별된 솔루션

## 2. JTBD — Jobs To Be Done (6-Part)

> 사용자가 이 도구를 "고용(hire)"하는 진짜 이유.

### Job Statement
**"여러 프로젝트와 현장 업무를 머릿속이 아닌 한 시스템에 담아, 마감을 놓치지 않고 끝내고 싶다."**

| Part | 내용 |
|------|------|
| **1. When (상황)** | 동시에 여러 프로젝트/거래처 건이 진행되고, 할 일이 머리·메모·카톡에 흩어져 있을 때 |
| **2. I want to (동기)** | 모든 할 일을 한 곳에 모아 우선순위와 마감을 한눈에 보고 싶다 |
| **3. So I can (기대 결과)** | 중요한 일을 놓치지 않고, 마감 전에 완료하며, "지금 뭘 해야 하지?" 고민을 없앤다 |
| **4. Functional job (기능적)** | 태스크 생성·분류·마감관리·진행추적, 현장 점검 결과 기록 |
| **5. Emotional job (감정적)** | "다 통제하고 있다"는 안정감, 놓친 게 없다는 확신 |
| **6. Social job (사회적)** | 거래처/현장에 "체계적으로 일하는 사람"으로 보이기 (점검 리포트 제출 등) |

**경쟁 대체재 (이 Job을 위해 현재 무엇을 고용 중인가)**: 엑셀, 카톡 나에게 보내기, 종이 메모, 노션, Trello

## 3. Lean Canvas — 신규 프로그램 2종

### 3-A. 현장 점검 체크리스트 앱 (추천 1순위)

| # | 항목 | 내용 |
|---|------|------|
| 1 | **Problem** | 소방/설비 현장 점검을 종이·엑셀로 하고, 사진/결과 취합·리포트화에 시간 소모. 누락 발생. |
| 2 | **Customer Segments** | 소방/설비 점검 실무자, 소규모 설비 시공·관리 업체 (Early adopter: 본인 + 동종 현장직) |
| 3 | **Unique Value Proposition** | "현장에서 모바일로 체크 → 즉시 PDF 리포트. 엑셀 취합 0분." |
| 4 | **Solution** | 점검 템플릿(체크리스트) → 항목별 OK/NG+사진+메모 → 리포트 자동 생성 → 거래처 공유 |
| 5 | **Channels** | 본인 현장 직접 사용 → 동료/협력업체 입소문 → 업종 커뮤니티 |
| 6 | **Revenue Streams** | (초기) 무료 개인용 / (확장) 팀 월 구독, 리포트 브랜딩 |
| 7 | **Cost Structure** | Supabase(무료~Pro), Vercel(무료~Pro), Storage(사진) — 거의 0원 시작 |
| 8 | **Key Metrics** | 주간 점검 완료 건수, 리포트 생성 수, 점검 1건 소요시간 |
| 9 | **Unfair Advantage** | 본인이 도메인(소방/설비) 실무자 — 현장 워크플로를 정확히 안다. 기성 SaaS가 못 따라옴. |

### 3-B. 견적/발주 관리 시스템 (추천 2순위)

| # | 항목 | 내용 |
|---|------|------|
| 1 | **Problem** | POSCO E&C·KOGAS 등 거래처 견적·발주 상태가 메일/엑셀에 흩어져 추적이 어려움 |
| 2 | **Customer Segments** | 견적·발주를 다루는 설비/시공 실무자, 소규모 업체 영업·구매 담당 |
| 3 | **UVP** | "견적부터 납품까지 칸반 한 보드에서 상태 추적" |
| 4 | **Solution** | 견적→발주→입고→정산 단계 칸반, 거래처/금액 필드, 첨부 |
| 5 | **Channels** | 본인 업무 → 사내/협력사 |
| 6 | **Revenue** | 개인 무료 → 소규모 팀 구독 |
| 7 | **Cost** | Supabase + Vercel + Storage |
| 8 | **Key Metrics** | 추적 중 발주 건수, 단계 전환 리드타임 |
| 9 | **Unfair Advantage** | 거래처 실무 경험 + 기존 칸반 엔진 재사용 |

> Lean Canvas 결론: 두 후보 모두 칸반 엔진 재사용 가능. **현장 점검**이 (a) 도메인 우위가 더 직접적이고 (b) 모바일 사진/체크리스트라는 차별점이 명확하며 (c) "엑셀 취합 0분"이라는 측정 가능한 가치가 뚜렷해 우위.

## 4. 사용자 페르소나

### 페르소나 A — "퇴근 후 정리하는 개인 사용자" (현재 핵심)
- **누구**: 본인. 여러 사이드/업무 프로젝트를 개인적으로 관리.
- **목표**: 흩어진 할 일을 한 곳에. 마감 놓치지 않기.
- **Pain**: 프로젝트가 늘수록 태스크 찾기 힘듦, 뭐가 급한지 안 보임.
- **핵심 니즈**: 전역 검색, 우선순위 시각화, 마감 알림.
- **사용 맥락**: 데스크톱, 짧고 잦은 세션.

### 페르소나 B — "현장의 점검 실무자" (확장 타깃)
- **누구**: 소방/설비 점검을 도는 실무자(본인 포함).
- **목표**: 현장에서 빠르게 점검·기록, 사무실 복귀 없이 리포트 제출.
- **Pain**: 종이/엑셀 이중 입력, 사진 따로 관리, 취합 시간, 항목 누락.
- **핵심 니즈**: 모바일 체크리스트, 사진 첨부, PDF 리포트.
- **사용 맥락**: 모바일/현장, 네트워크 불안정 가능.

### 페르소나 C — "발주·정산 챙기는 거래처 담당" (확장 타깃)
- **누구**: 거래처(POSCO E&C/KOGAS 등) 대상 견적·발주 처리자.
- **목표**: 어떤 건이 어느 단계인지, 누락·지연 없이 추적.
- **Pain**: 견적/발주가 메일·엑셀에 분산, 상태 파악 수고.
- **핵심 니즈**: 단계별 보드, 거래처/금액 필드, 첨부, 검색.
- **사용 맥락**: 데스크톱 위주.

## 6. 우선순위 매트릭스 (임팩트 vs 구현 난이도)

> 임팩트 = 일일 사용/완료율 기여. 난이도 = 현재 스택 기준 구현 비용. 스키마 기존 필드 존재 여부 반영.

### pjt_manager 추가 기능

| 기능 | 임팩트 | 난이도 | 사분면 | 비고 |
|------|:------:|:------:|--------|------|
| 전역 태스크 검색 | 높음 | 중 | **Quick Win (우선)** | 모든 프로젝트 통합 — 다중 프로젝트의 핵심 통증 해소 |
| 우선순위 배지+필터 | 높음 | **낮음** | **Quick Win (최우선)** | `priority` 필드 이미 존재 → UI만 |
| 마감일 탭 알림 배지 | 중~높음 | **낮음** | **Quick Win** | document.title 조작, 비용 0 |
| 칸반→리스트 뷰 | 중 | 중 | 검토 | 검색과 시너지 |
| 주간 리포트 | 중 | 중 | 검토 | 리텐션 보조 |
| 일괄 선택/이동 | 중 | 중 | 검토 | 파워유저용 |
| 파일 첨부(Storage) | 중 | 중~높 | 검토 | 현장점검 모듈에서 더 가치 큼 |
| 반복 태스크 | 중 | 중~높 | 후순위 | 규칙 엔진 필요 |
| AI 태스크 분해 | 중 | 높음 | 후순위(차별화) | Claude API, 비용 발생 |
| 댓글/활동 로그 | 낮~중 | 중 | 후순위 | 1인 사용 시 가치 낮음 |

### 신규 프로그램

| 후보 | 임팩트(도메인 적합) | 난이도(엔진 재사용도) | 평가 |
|------|:----:|:----:|------|
| **현장 점검 체크리스트** | 높음 | 중 (체크리스트+Storage 재사용) | **1순위** |
| 견적/발주 관리 | 중~높 | 낮~중 (칸반 그대로) | 2순위 |
| 거래처 CRM | 중 | 중 | 3순위 (발주와 결합 시 가치↑) |
| 문서/도면 관리 | 낮~중 | 중~높 (버전관리 부담) | 후순위 |

## 7. Beachhead Segment & GTM (Go-To-Market)

### Beachhead (Geoffrey Moore — 교두보 세그먼트)
**"소방/설비 현장 점검을 직접 도는 1인 실무자(본인 포함)와 소규모 협력업체"**

| 4-기준 | 평가 |
|--------|------|
| 도달 가능성 | 높음 — 본인이 현장에 있고 동종 네트워크 보유 |
| 지불 의향/통증 강도 | 높음 — 엑셀 취합·누락은 실제 시간/리스크 손실 |
| 경쟁 공백 | 높음 — 기성 SaaS는 비싸고 무거움, 개인용 공백 |
| 입소문 가능성 | 높음 — 현장직 커뮤니티는 좁고 추천 영향 큼 |

### GTM
- **Phase 0 (Dogfooding)**: 본인 현장 실제 점검에 사용 → 워크플로 검증, "엑셀 대비 시간 단축" 수치 확보.
- **Phase 1 (입소문)**: 동료/협력업체 2~3명에게 리포트 공유 → 무료로 확산.
- **채널**: 직접 사용 → 현장 동료 → 업종 커뮤니티/오픈채팅.
- **핵심 지표**: 주간 점검 완료 건수, 리포트 생성 수, 점검 1건 소요시간(엑셀 대비), 재사용율(W2 리텐션).

## 8. 최종 추천 및 근거

### pjt_manager 추가기능 TOP 3

1. **우선순위 배지 + 필터 (최우선)**
   - 근거: 임팩트 높음 + 난이도 최저. `Task.priority` 필드가 **이미 스키마에 존재**(`'low'|'normal'|'high'|'urgent'`)하므로 백엔드 작업 없이 UI 배지/필터만 추가하면 됨. ROI 최고. OST Opportunity 2 직접 해소.

2. **전역 태스크 검색**
   - 근거: 다중 프로젝트 사용자의 1순위 통증(OST Opportunity 1). 프로젝트가 늘수록 가치가 커지는 구조적 기능 → 리텐션 직결. 제목·설명·태그 통합 검색.

3. **마감일 탭 알림 배지**
   - 근거: 난이도 최저(document.title), 비용 0, "마감 놓침"(OST Opportunity 3) 즉시 완화. 앱을 안 보고 있어도 능동 알림 → 일일 복귀 유도.

> 권장 구현 순서: 우선순위 배지 → 전역 검색 → 알림 배지 (난이도/의존성 기준)

### 신규 프로그램 TOP 1

**현장 점검 체크리스트 앱**
- 근거 (3중 정렬):
  1. **도메인 우위 (Unfair Advantage)**: 본인이 소방/설비 실무자 → 기성 SaaS가 못 따라오는 현장 워크플로 이해.
  2. **엔진 재사용**: 기존 체크리스트·태스크·Storage 구조를 점검 항목/사진/리포트로 확장 가능 → 개발 비용 절감.
  3. **측정 가능한 가치**: "엑셀 취합 0분, 누락 0건"이라는 명확한 통증 해소 + Beachhead 4기준 전 항목 우수.
- 견적/발주 관리는 강력한 2순위 — 현장 점검으로 도메인 사용자 확보 후 같은 칸반 기반으로 자연 확장 가능.
